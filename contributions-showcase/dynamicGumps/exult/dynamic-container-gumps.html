Overview

Dynamic Container Gumps provide an advanced container display system for Exult that extends the standard Container_gump class. This system enables:

  • Snap Zone Support: Guided item placement for ritual containers and complex item arrangements
  • Debug Visualization: Visual tools for designers to verify placement zones and container boundaries
  • Automatic Selection: Gump Manager automatically uses Dynamic_container_gump when configuration is detected
  • Backward Compatibility: Standard containers without configuration continue using existing Container_gump

Initialization Process

When a container gump is created, the Gump Manager performs configuration detection and initialization:

1
Configuration Check
Gump Manager checks if the gump shape has container_area configuration in gump_info.txt
2
Gump Selection
If configuration exists, Dynamic_container_gump is instantiated; otherwise uses standard Container_gump
3
Load Container Area
Reads container_area section to set object placement boundaries and dimensions
4
Load Snap Zones
Parses all snap zone definitions for this gump shape from snap_zones section
5
Initialize Debug Settings
Reads debug flags bitmask to determine which visualization features to enable

Snap Zones System

Snap zones enable guided item placement within containers. This is essential for complex rituals or arrangements where items must be positioned at specific locations (similar to Ultima 8's sorcery system).

How Snap Zones Work

  1. Multiple zones are defined within a container, each with boundaries and a target snap position
  2. When an item is dropped, the system determines which zone contains the drop position
  3. If a zone is found, the item automatically snaps to that zone's target coordinates
  4. If multiple zones overlap, zone resolution uses priority and geometric algorithms
  5. Items without a matching zone remain at their drop position

Zone Configuration Format

Snap zones are defined in gump_info.txt under the snap_zones section:

%%section snap_zones
# Format: :shape/zone_id/zone_x/zone_y/zone_w/zone_h/snap_x/snap_y/priority/zone_type
:111/pz_candle/20/0/20/20/25/5/1/candle
:111/locus/55/55/20/20/65/65/3/focus
:111/component1/0/55/20/20/5/65/2/component
%%endsection

Zone Properties

Property Description
zone_id Unique identifier for this zone
zone_x/y Top-left corner of zone (relative to container)
zone_w/h Width and height of zone boundary
snap_x/y Target position where item snaps
priority Resolution priority (higher = more important)
zone_type Zone category identifier

Priority Resolution

When items are dropped in overlapping zones:

  • Different priorities: Highest priority zone wins
  • Equal priorities: Axis-aligned geometric split determines ownership

This hybrid approach ensures predictable behavior while allowing complex zone layouts.

Debug Visualization

Debug features help designers verify container setup and zone placement. They're controlled by an 8-bit bitmask in the container_area configuration.

Debug Flags Reference

Flag Name Hex Value Description
GUMP_DEBUG_BORDER 0x01 Draw container area border
GUMP_DEBUG_GRID_MAJOR 0x02 Draw major grid lines (10px spacing)
GUMP_DEBUG_GRID_MINOR 0x04 Draw minor grid lines (5px offset)
GUMP_DEBUG_SNAP_BORDER 0x08 Draw snap zone boundaries
GUMP_DEBUG_SNAP_CROSS 0x10 Draw snap zone target crosshairs
GUMP_DEBUG_CONSOLE 0x20 Log debug messages to console
GUMP_DEBUG_ALL 0x3F Enable all debug features

Configuration Examples

Development (all debug enabled):

%%section container_area
:111/30/20/130/130/63    # 63 = 0x3F (all flags)
%%endsection

Production (no debug):

%%section container_area
:111/30/20/130/130/0     # 0 = no debug features
%%endsection

Selective debugging (border + grid only):

%%section container_area
:111/30/20/130/130/3     # 3 = 0x01 | 0x02 (border + grid)
%%endsection

Configuration System

All Dynamic Container Gump configuration is stored in gump_info.txt, which already exists in Exult. This choice ensures:

  • ✓ Infrastructure already established and tested
  • ✓ File parsing system already in place
  • ✓ Consistent with Exult's naming conventions (*_info.txt)
  • ✓ No new file handling or dependencies required

Key Features & Implementation

1. Automatic Gump Selection

The Gump_manager automatically instantiates Dynamic_container_gump when configuration is detected.

// In Gump_manager.cc, add_gump() function
if (Dynamic_container_gump::has_config(shapenum)) {
    auto* dgump = new Dynamic_container_gump(cont, px, py, shapenum, false);
    // ... initialize and return
}

2. Method Overrides

add()

Implements snap zone logic and extensive debug logging for item placement.

paint()

Adds debug grid overlay visualization on top of standard rendering.

find_snap_zone()

Finds the best snap zone for a given relative position using priority and geometry.

clone()

Enables proper gump cloning with configuration inheritance.

4. Design Philosophy

All changes follow Exult's design patterns:

  • Lightweight: Minimal code additions, maximum reuse of existing systems
  • Extensible: Uses inheritance rather than parallel alternatives
  • Backward Compatible: Containers without configuration unchanged
  • Consistent: Follows existing Exult API patterns (get_owner(), find_gump(), etc.)

Usecode Integration

Two new intrinsic functions enable programmatic access to item positions within containers:

get_item_gump_position()

Returns: [x, y] array with item's gump coordinates

Returns empty array if: Item invalid, not in container, or gump not open

var pos = UI_get_item_gump_position(item);
if (pos) {
    var x = pos[0];
    var y = pos[1];
}

set_item_gump_position()

Returns: 1 on success, 0 on failure

Fails if: Invalid item, not in container, gump not open, or negative coords

var success = UI_set_item_gump_position(
    item, 65, 65
);
